# TabOrganizer · 完整說明

> Chrome / Edge MV3 瀏覽器分頁整理器。一鍵把所有分頁分類、HEAD 檢查 404、截圖保留 URL 與標題，然後關閉分頁釋放記憶體；管理頁以縮圖卡片網格瀏覽，可按原分類一鍵重新打開。

## 1. 專案目標

長時間使用瀏覽器累積大量分頁是常見痛點：占用記憶體、難以快速回顧曾經看過什麼、想關掉又怕丟失。TabOrganizer 透過下列流程解決：

```
分頁 → 分類建議 → HEAD 檢查 → 截圖歸檔 → 關閉釋放記憶體 → 縮圖管理頁瀏覽 → 需要時一鍵重開
```

設計取捨：
- **保留可回溯性**：URL + 標題 + 縮圖 + 分類 + 時間都存下來，不會「就此消失」。
- **積極釋放記憶體**：截圖完就關閉分頁，不留在背景。
- **自動剔除無效連結**：截圖前 HEAD 檢查，404 / 失效不浪費截圖成本。
- **黑白灰極簡風格**：嚴格遵守 ii Design Language，視覺乾淨不干擾。

## 2. 核心功能

### 2.1 分頁分類
- 使用者手動建立分類（如「工作」、「學習」、「娛樂」、「待分類」）
- 每個分類可設定 domain 規則，支援 wildcard：
  - `github.com` — 完全匹配
  - `*.example.com` — 比對子網域
- 掃描分頁時自動比對規則並推薦分類；使用者可在管理頁手動改分類

### 2.2 截圖保留 URL 並關閉分頁
- 按下「一鍵整理」後序列化處理每個分頁：
  1. `chrome.tabs.update({ active: true })` 切換到分頁
  2. 等 250 ms 讓畫面 render
  3. `chrome.tabs.captureVisibleTab` 截圖（JPEG q80）
  4. 縮圖（512 px WebP）存 IndexedDB，原圖透過 `chrome.downloads` 存到 `下載資料夾/TabOrganizer/<domain>/`
  5. 寫入 Dexie 資料表
- 全部處理完後 `chrome.tabs.remove` 關閉所有分頁

### 2.3 一鍵整理所有分類
- popup 與 manager 都有「一鍵整理」按鈕
- 服務協調器（service worker）執行完整流程，進度透過 `chrome.runtime.sendMessage` 推送
- manager 頁面接收進度訊息，全螢幕 overlay 顯示 stage、進度條、目前處理的標題與最終總結

### 2.4 自動剔除 404
- 截圖前先對所有 URL 並行發 HEAD 請求（concurrency 8、timeout 5 s）
- 若回 405 / 403（部分網站封 HEAD），自動 fallback 改用 `GET` + `Range: bytes=0-0`
- 失效（HTTP ≥ 400 / timeout / network error）的不截圖、寫入「已剔除清單」，分頁仍會關閉

### 2.5 附加功能（使用者選用）
| 功能 | 說明 |
|---|---|
| 搜尋總歸檔 | 在 manager header 的搜尋框輸入關鍵字，即時 filter 標題 / URL / 分類 |
| 定期重掃舊連結 | 設定頁可選「每小時 / 每日 / 每週 / 每月 / 自訂分鐘」；背景任務透過 `chrome.alarms` 對所有歸檔再跑一次 HEAD，失效項目標紅色狀態點 |
| 按原分類重新打開 | sidebar hover 分類顯示「重開」按鈕，一鍵把該分類所有歸檔 URL 用新分頁重新打開 |

## 3. 快速開始

### 3.1 前置需求
- Node.js 18+（建議 20 LTS）
- npm 10+
- Chrome 116+ 或 Edge 116+（minimum_chrome_version 設於 manifest）

### 3.2 安裝與建置
```powershell
cd D:\Desktop\AI\MiniPrograms\TabOrganizer
npm install
npm run build
```

`dist/` 資料夾即為可載入的擴充功能。

### 3.3 載入到瀏覽器
1. 開 `chrome://extensions/`（Edge 是 `edge://extensions/`）
2. 右上開啟「開發人員模式」
3. 點「載入未封裝項目」
4. 選 `D:\Desktop\AI\MiniPrograms\TabOrganizer\dist`
5. 工具列出現 TabOrganizer icon（黑底白色三條膠囊）

### 3.4 第一次使用
1. 點工具列 icon → 開啟 popup
2. 點「分類設定」→ 建立第一個分類（例：工作 + 規則 `github.com`）
3. 隨意開幾個分頁（最好混入 GitHub 與其他網站）
4. 回到 popup 看到「目前可整理分頁 N / 已建議分類 M / 未分類 K」
5. 點「一鍵整理所有分頁」
6. 自動跳到管理頁，看見進度條 → 完成後縮圖網格出現
7. 點任一縮圖看詳情、按「重新開啟」即可在新分頁打開原 URL

## 6. 視覺風格

嚴格遵守 `D:\Desktop\AI\visual-style-docs` 的 **ii Design Language**。

### 6.1 八條鐵則
1. **黑白灰配色**——介面只用白 / 灰 / 黑，狀態 dot 是唯一彩色例外
2. **0.5 px 邊框**——比 1 px 更精緻；Retina 顯示器下為實際半物理像素
3. **10 px 主圓角**——卡片 / 按鈕 / input；12 px 用於最外層 shell；4 px 用於 tag / tooltip
4. **8 / 16 px 標準 padding**——按鈕 `8px 16px`、input `8px 12px`、section header `12px 16px`
5. **15-25 ms 過渡**——`--t-fast: 0.15s`、`--t-base: 0.2s`
6. **Inter 字體**——`Inter → Microsoft JhengHei → PingFang TC → Noto Sans TC`
7. **絕不用 emoji**——介面文字、按鈕標籤、提示一律不用
8. **Scrollbar 3-6 px**——細而不搶眼

### 6.2 字級階梯
| Token | px | 用途 |
|---|---|---|
| `--text-micro` | 9 | label-micro（大寫小標） |
| `--text-xs` | 10 | tooltip、tag、表格 header |
| `--text-sm` | 11 | 次標籤、輔助文字 |
| `--text-base` | 12 | input、button |
| `--text-md` | 13 | 主文本 |
| `--text-lg` | 14 | 區段標題、brand |
| `--text-xl` | 16 | 頁面主標題 |

### 6.3 狀態色（唯一彩色例外）
| Token | 色 | 用途 |
|---|---|---|
| `--status-ok` | `#00ff41` | 有效連結、可整理 |
| `--status-warn` | `#facc15` | 整理中（脈動動畫） |
| `--status-err` | `#e53935` | 失效連結 |
| `--status-idle` | `#c8c6c6` | 未啟用、待分類 |

### 6.4 視覺元件清單
- `.ii-card` — 白底 + 0.5 px 邊框 + 10 px 圓角
- `.glass-box` — 78% 白 + blur(20px) + 0.5 px 邊框
- `.btn` / `.btn-primary` / `.btn-danger` / `.btn-ghost`
- `.input` — 8 × 12 padding，focus 變黑邊
- `.label-micro` — Space Grotesk mono 9 px uppercase + 0.08 em letter-spacing
- `.status-dot.ok/warn/err/idle` — 6 × 6 px + glow box-shadow
- `.tag` — pill 形 badge，bg `--bg-surface`
- `.has-tooltip` + `.tooltip-content` — hover 顯示黑底白字提示

## 8. 技術架構

### 8.1 技術棧
| 層 | 選擇 | 為何 |
|---|---|---|
| 語言 | TypeScript 5.6（strict） | 與其他 MiniPrograms 一致 |
| UI | Vue 3.5（`<script setup>`） | 與其他 MiniPrograms 一致 |
| 建置 | Vite 5 + `@crxjs/vite-plugin` 2.x | MV3 + Vite 唯一成熟方案；自動處理 manifest 路徑改寫、HMR |
| 狀態 | Pinia 2 | popup 與 manager 各自獨立 store；跨 context 同步靠 `chrome.storage` + `chrome.runtime.sendMessage` |
| IndexedDB | Dexie.js 4 | 避免原生 IDB 在 service worker 短壽命下的 schema migration 樣板 |
| HEAD 檢查 | 原生 `fetch` | 不需額外依賴 |

### 8.2 三大執行 context
Chrome MV3 擴充功能有三個獨立的執行環境：

```
┌─────────────────────┐   sendMessage   ┌──────────────────────┐
│ popup（短壽命）     │ ─────────────→  │ service worker        │
│ Vue + Pinia         │ ←─────────────  │ 協調器 + 排程         │
└─────────────────────┘                 └──────────────────────┘
                                                  │
                                                  │ sendMessage
                                                  ▼
                                        ┌──────────────────────┐
                                        │ manager.html（長壽命）│
                                        │ Vue + Pinia + Dexie  │
                                        └──────────────────────┘
```

- **popup**：點工具列 icon 開啟，關閉就銷毀；只做輕量掃描與按鈕分派
- **service worker**：背景常駐（但 Chrome 會 unload 閒置的 SW，所以任務要 chunk 化）；負責一鍵整理協調、定期重掃 alarm、跨 context 訊息路由
- **manager.html**：獨立分頁，承載大部分 UI；訂閱 service worker 的進度訊息

### 8.3 一鍵整理執行序列
```
[popup] click「一鍵整理」
   │
   └─→ chrome.runtime.sendMessage({ type: 'organize/start' })
            │
            └─ [service-worker]
                  │ ① chrome.storage.local.set({ organizeInProgress: true })
                  │ ② 開啟 manager（如果還沒開）並導引到進度視圖
                  │ ③ tabScanner.scanAllTabs() → TabCandidate[]
                  │ ④ broadcast({ type: 'organize/progress', stage: 'scanning' })
                  │ ⑤ linkChecker.checkBatch(urls, { concurrency: 8 })
                  │    → 對死連結寫入 ExcludedRecord
                  │    → broadcast stage: 'checking'
                  │ ⑥ for each alive candidate（序列化）：
                  │    a. chrome.tabs.update(id, { active: true })
                  │    b. await sleep(250)
                  │    c. chrome.tabs.captureVisibleTab → dataUrl
                  │    d. resizeToThumb → Blob (512px webp)
                  │    e. db.archives.add(record)
                  │    f. chrome.downloads.download(原圖)
                  │    g. broadcast({ stage: 'snapshotting', current, total, title })
                  │ ⑦ chrome.tabs.remove(allIds)
                  │ ⑧ broadcast({ type: 'archive/changed' })
                  │ ⑨ broadcast({ type: 'organize/done', summary })
                  │ ⑩ chrome.storage.local.set({ organizeInProgress: false })
                  └─→ [manager] OrganizeProgressOverlay 顯示 summary
                                useArchiveStore.refresh() 拉新資料
```

100 個分頁估算耗時：`(250 ms 切換 + ~200 ms 截圖 + ~50 ms 寫入) × 100 ≈ 50 秒`。

## 9. 核心模組

### 9.1 `src/modules/tab-scanner.ts`
```ts
scanAllTabs(): Promise<ScanResult>
```
- `chrome.tabs.query({})` 拿所有分頁
- 過濾掉 `chrome://`、`edge://`、`chrome-extension://`、`about:`、`devtools://`、`view-source:`
- 每個分頁取 domain（去 `www.`、lowercase）
- 呼叫 `categoryEngine.suggest(domain)` 取推薦分類
- 回傳 `{ candidates, groupedByDomain, total, suggested, unassigned }`

### 9.2 `src/modules/category-engine.ts`
```ts
suggestCategoryId(domain, state): number | null
```
- 先檢查精確匹配（`domain === rule.domain`）
- 再檢查 wildcard（`*.example.com` → 比對 `example.com` 與 `*.example.com`）
- 內建 cache，避免每次掃描都重打 DB；`invalidateCategoryCache()` 在編輯分類後清除

### 9.3 `src/modules/link-checker.ts`
```ts
checkBatch(urls, { concurrency: 8, timeoutMs: 5000 }): Promise<Map<url, CheckResult>>
```
- Worker pool 模式（不是 Promise.all 所有 URL 一次發），用一個共享 index 變數
- 每個請求 `AbortController` 控制 timeout
- HEAD 失敗且 status 為 405 / 403 時自動 fallback 改 GET（`Range: bytes=0-0` 只取第一個 byte）
- **CORS 緩解**：service worker 內 fetch 不受 CORS 限制（需 `host_permissions: ["<all_urls>"]`），所以 LinkChecker 必須在 service worker 跑，不能在 popup / manager

### 9.4 `src/modules/tab-snapshotter.ts`
```ts
snapshotTabs(tabs, onProgress?): Promise<{ archivedCount, failedCount }>
```
- 序列化處理（不能並行，`captureVisibleTab` 只能截目前 active tab）
- 每個 tab：切 active → sleep 250 ms → captureVisibleTab → resize 縮圖 → 寫 Dexie → chrome.downloads
- onProgress 回呼用於推送 service worker 的 broadcast
- try/catch 包住每個 tab，失敗的記到 console + failedCount++，不影響後續

### 9.5 `src/modules/archive-store.ts`
Dexie schema：
```ts
db.version(1).stores({
  archives:   '++id, url, domain, categoryId, archivedAt, status, [categoryId+archivedAt]',
  categories: '++id, name, createdAt',
  domainRules:'++id, domain, categoryId',
  excluded:   '++id, url, domain, excludedAt',
  scanHistory:'++id, ranAt',
});
```
- `thumbBlob: Blob` 不建索引
- `[categoryId+archivedAt]` 複合索引給「分類內按時間排序」用
- `deleteCategory(id)` 用 transaction：刪分類 + 刪相關規則 + 把該分類的歸檔改為 `categoryId: null`

### 9.6 `src/modules/search-engine.ts`
```ts
searchArchives(archives, categories, query): ArchivedTab[]
```
- 把 query 拆 token（空白分隔）
- 每個歸檔組合 `[title, url, domain, categoryName]` 為 hay
- 每個 token 都要在 hay 出現才算 match（AND）

### 9.7 `src/modules/periodic-revalidator.ts`
```ts
ensureAlarmConfigured(intervalMinutes): Promise<void>
runRevalidation(): Promise<{ checked, gone }>
```
- `chrome.alarms.create('tab-organizer-revalidate', { periodInMinutes })`
- 觸發時對所有歸檔 URL 跑 `checkBatch`
- 失效項目把 `status` 改為 `'gone'`，`lastCheckedAt` 與 `lastStatusCode` 更新
- 預設 7 天（`60 × 24 × 7` 分鐘），可在設定頁改

## 10. Chrome 權限說明

manifest.json 宣告的權限與用途：

| 權限 | 為何需要 |
|---|---|
| `tabs` | 列舉所有分頁（標題、URL、windowId）、切換 active、關閉 |
| `activeTab` | `captureVisibleTab` 需要 |
| `downloads` | 把原圖（dataURL）寫到使用者下載資料夾 |
| `storage` | `chrome.storage.local` 存設定（重掃頻率、整理中旗標） |
| `alarms` | 排程定期重掃 |
| `tabGroups` | 為未來「讀現有 Tab Groups 作為分類來源」預留 |
| `host_permissions: ["<all_urls>"]` | HEAD 檢查任意網站 + captureVisibleTab 對任意網域 |

**為什麼需要 `<all_urls>` host permissions**：
- `chrome.tabs.captureVisibleTab` 需要該分頁的 host permission，否則回傳空 dataURL
- `fetch` 在 service worker 內發 cross-origin HEAD 請求需要 host permission

這個權限範圍較大，但是 TabOrganizer 不會把任何資料傳到外部，所有資料都在使用者本機（IndexedDB + 下載資料夾）。

## 11. 資料儲存

### 11.1 IndexedDB（`TabOrganizerDB`）
| 表 | 欄位 | 大小估算 |
|---|---|---|
| `archives` | id, url, title, domain, categoryId, **thumbBlob**, downloadId, downloadPath, archivedAt, status, lastCheckedAt, lastStatusCode | 縮圖 WebP q80 ≈ **30 KB/張**；1000 張 ≈ 30 MB |
| `categories` | id, name, color, createdAt | 可忽略 |
| `domainRules` | id, domain, categoryId | 可忽略 |
| `excluded` | id, url, title, domain, reason, statusCode, excludedAt | 可忽略 |
| `scanHistory` | id, ranAt, totalChecked, totalGone | 可忽略 |

IndexedDB 預設配額視瀏覽器與磁碟空間而定，通常給單一 origin 的配額是磁碟可用空間的 60% 以上，幾十 MB 不會踩到上限。

### 11.2 chrome.storage.local
僅存設定（小）：
```json
{
  "tabOrganizer.settings": {
    "revalidateIntervalMinutes": 10080,
    "organizeInProgress": false
  }
}
```

### 11.3 下載資料夾
原圖以 JPEG 存於：
```
<使用者下載資料夾>/TabOrganizer/<slugified-domain>/<timestamp>-<slugified-title>.jpg
```
例：
```
C:/Users/<you>/Downloads/TabOrganizer/github-com/2026-05-20T14-23-45-789Z-vue-3-5-issue-9210.jpg
```

如要備份或上傳到雲端，這個資料夾是長期保存的本體；IndexedDB 縮圖只是快速顯示用的快取。

## 13. 常見問題

### Q1：載入到 Chrome 後 popup 點開是空白？
查 `chrome://extensions/` 該擴充功能的「錯誤」按鈕。最常見原因：
- `dist/` 不完整 → 重新 `npm run build`
- Chrome 版本 < 116 → 升級瀏覽器

### Q2：整理時截圖有幾張是空白/黑色？
原因可能是：
- 該分頁是 PDF（Chrome 對 PDF 的 captureVisibleTab 行為不一致）
- 該分頁尚未載入完成（被預設 250 ms 等待時間蓋不住）
- 解法：建立 lazy 載入的分頁先讓它完整跑完再整理；或在 [src/modules/tab-snapshotter.ts](src/modules/tab-snapshotter.ts) 把 `SETTLE_DELAY_MS` 調大（如 500）

### Q3：所有分頁都被剔除說 404？
- 確認瀏覽器有網路
- 檢查公司網路是否封鎖某些 domain
- 查 service worker console（`chrome://extensions/` → 該擴充功能 → 「service worker」連結）看 fetch error
- 如果是 CORS 相關，確認 `host_permissions: ["<all_urls>"]` 還在

### Q4：整理流程一直卡在「掃描分頁中」？
- service worker 可能被 Chrome unload。打開 `chrome://extensions/` 找到該擴充功能點「service worker」喚醒它
- 或直接重新點「一鍵整理」

### Q5：管理頁打開很慢？
- 100 張以上歸檔時，初次載入 Blob URL 需要時間
- 可以在 [src/components/archive/ThumbnailGrid.vue](src/components/archive/ThumbnailGrid.vue) 加 IntersectionObserver lazy load

### Q6：定期重掃沒在跑？
- 確認設定頁不是「關閉」
- Chrome 對 background alarm 有節流；最低 1 分鐘間隔
- 查 `chrome://extensions/` → service worker → console，搜 `revalidate`

### Q7：要把資料搬到另一台電腦？
目前沒有匯出 UI（規劃時你沒選這項）。臨時方案：
1. 開 manager 頁 → 開 DevTools → Application → IndexedDB → `TabOrganizerDB`
2. 全選 `archives` 表 → 複製為 JSON
3. 在新電腦的 DevTools 一樣的位置 import

或從下載資料夾 `TabOrganizer/` 直接複製過去（保留原圖；縮圖會丟失，但隨時可以從原圖重生）。

## 14. 已知限制

| 限制 | 原因 | 影響 |
|---|---|---|
| 100 分頁約需 50 秒 | `captureVisibleTab` 只能截 active tab，必須序列化 | 整理中不能用瀏覽器 |
| Service worker 可能中途被 unload | Chrome MV3 設計 | 大批整理時可能要重點「一鍵整理」續跑 |
| 無法截 chrome:// / chrome-extension:// | Chrome 安全限制 | 這些分頁直接過濾掉，不關閉 |
| PDF 截圖品質不一 | captureVisibleTab 對 PDF 行為有限 | PDF 縮圖可能空白，但 URL 仍保留 |
| 不支援 Firefox / Safari | MV3 + Chromium 專屬 API | 規劃時已確認只做 Chrome / Edge |
| 縮圖大小固定 512 px | 在 [src/modules/tab-snapshotter.ts](src/modules/tab-snapshotter.ts) 的 `THUMB_MAX_WIDTH` | 可改但要重新 build |

## 15. 後續可擴充方向

依使用者反饋優先度排序的可選擴充：

1. **匯出 / 匯入 JSON**——讓使用者備份或跨電腦同步歸檔
2. **AI 自動分類**——用 Claude API 根據標題 + 摘要推薦分類（需要 API key 設定）
3. **群組整理**——只整理「閒置超過 N 天」的分頁
4. **Tab Groups 整合**——讀現有 Chrome Tab Groups 作為分類來源
5. **批次操作**——管理頁多選縮圖一次刪除 / 改分類 / 重開
6. **OCR 搜尋**——對截圖跑 OCR，把文字也納入搜尋索引
7. **匯出整理報告**——產生 PDF / HTML 報告列出本次關掉哪些分頁、釋放多少 MB
8. **鍵盤快捷鍵**——`Cmd+Shift+T` 一鍵整理、`Cmd+K` 開搜尋
